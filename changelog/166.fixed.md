gNMI transport security is now configurable instead of hardcoded plaintext: `GNMI_TLS_MODE` (`insecure` | `skip-verify` | `ca-cert` with `GNMI_CA_CERT`) controls every gNMI connection, and a malformed mode fails loud rather than falling back to plaintext. This matters beyond hardening — containerlab's default SR Linux gNMI server is TLS-only, so the previous baked-in `insecure=True` could not connect to a freshly deployed lab at all. Device credentials are now resolved from the worker environment (`GNMI_USERNAME`/`GNMI_PASSWORD`) inside activities instead of being accepted as activity arguments, which Temporal persists unencrypted in workflow history; a guard test fails any future activity that accepts credential-named parameters. The rollback e2e test now runs against a default containerlab node with no manual gNMI listener setup.
